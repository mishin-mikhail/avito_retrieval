# Поиск статей справки Авито по пользовательскому запросу

Retrieval-этап RAG-пайплайна: по тексту вопроса пользователя вернуть
ранжированный топ-10 статей справочного центра. Метрика — MAP@10.

**Результат: MAP@10 = 0.5588 (test), 0.6499 (calibration, честная 5-fold оценка).**

## Подход

Гибрид из пяти каналов ранжирования, слитых через weighted Reciprocal Rank
Fusion, с доводкой порядка cross-encoder реранкером:

| Канал | Реализация | Зачем |
|---|---|---|
| A. Лексический | BM25 (rank-bm25), Snowball-стемминг, title с бустом ×3 | точные совпадения по редким терминам («СБП», «пункт выдачи») |
| B. Символьный | TF-IDF по char 3–5-граммам (sklearn) | устойчивость к опечаткам («оюувь», «балы») |
| C. Семантический | bi-encoder `intfloat/multilingual-e5-large`; статьи → чанки по 160 слов (перекрытие 40) с title в каждом чанке; скор статьи = max по чанкам | разрыв «язык пользователя ↔ язык статьи» |
| D. kNN по выданной разметке calibration | скор статьи = близость запроса (e5-эмбеддинги) к размеченным организаторами запросам calibration, у которых она в ground_truth; классический kNN поверх выданных данных | конвенции разметки «тип вопроса → статья», невыводимые из текстов |
| E. Семантический-2 | bi-encoder `BAAI/bge-m3`, та же схема чанков | второй энкодер ошибается иначе, ансамблевое разнообразие |

Слияние: `score = Σ w_i / (40 + rank_i)`, веса `[1.0, 2.5, 0.5, 5.0, 2.0]`
(BM25, e5, char, kNN, bge-m3).
Затем топ-15 кандидатов мягко переранжируются `BAAI/bge-reranker-v2-m3`
(вход — запрос + лучший чанк статьи): финальный скор = RRF-комбинация позиции
слияния и позиции по скору реранкера с весом w_ce=0.5. Жёсткая пересортировка
чистым реранкером проверялась и оказалась хуже: CE не знает конвенций разметки,
которые несёт канал D.

## Обработка данных

**HTML статей**: парсинг BeautifulSoup (lxml), удаляются только `script`/`style`;
текст извлекается с сохранением содержимого таблиц, спойлеров и вкладок — в них
часто находятся сами инструкции. Пробелы и переносы схлопываются. Код —
`src/common.py::clean_html`.

**Документ для лексических каналов (A, B)**: `title × 3 + чистый текст body` —
трёхкратный повтор заголовка работает как буст его слов в BM25/TF-IDF.
Токенизация для BM25: нижний регистр, «ё»→«е», токены `[а-яa-z0-9]+`,
Snowball-стеммер для русского.

**Документ для семантических каналов (C, E)**: статья режется на перекрывающиеся
чанки по 160 слов (шаг 120), заголовок вшивается в каждый чанк
(`"{title}. {фрагмент}"`); скор статьи = максимум близости по её чанкам. Это
решает две проблемы: длинные статьи (в среднем ~5 тыс. символов) не размывают
релевантный абзац, а короткие статьи представлены хотя бы заголовком.

## Использование calibration и отсутствие утечек

- Канал D использует выданную разметку на calibration как обучающие
  данные для классического kNN. Никакая
  дополнительная разметка не производилась: ни calibration, ни тем более test
  вручную не размечались, тестовая разметка не восстанавливалась, ответы под
  конкретные query_id не задавались — канал D является общим алгоритмом и работает
  для любого нового запроса.
- Все локальные оценки — с 5-fold маскированием: при скоринге запроса из
  calibration его собственный фолд исключается из соседей kNN, иначе оценка
  была бы завышена. Для test-инференса используется вся calibration; разметка
  test неизвестна и нигде не используется.
- BM25 и char-TF-IDF обучаются только на текстах статей (не на запросах).
- Гиперпараметры подбирались на calibration: сетка на train-части (350
  запросов), контроль переобучения на holdout (150).

## Эволюция решения

| Итерация | MAP@10 (calibration) |
|---|---|
| BM25 + стемминг + чистка HTML | 0.2587 |
| + bi-encoder e5-base, слияние RRF | 0.3715 |
| e5-base → e5-large, + канал char n-грамм | 0.4491 |
| + cross-encoder реранкер (топ-20) | 0.4756 |
| + kNN-канал по разметке, мягкий бленд реранкера | 0.6313 |
| тюнинг весов (w_knn 3→5, RRF_K 60→40) + канал bge-m3, бленд depth=15/w_ce=0.5 | 0.6499 |

На тесте: первая отправка (конфигурация 0.6313) — **0.5543**; финальная — **0.5588**.
Разрыв calibration/test объясняем почти-дублями запросов внутри calibration,
которые делают 5-fold оценку kNN-канала оптимистичной.

Проверенные и отвергнутые идеи: эвристика «статей-близнецов» (дубли title для
сайта/приложения; −0.003), жёсткая пересортировка реранкером (−0.09 после
появления канала D), агрегация kNN по топ-2/3 соседям вместо максимума
(хуже на train и holdout).

## Анализ ошибок: найденные типы и принятые меры

Каждая итерация начиналась с разбора худших запросов по AP@10 на calibration
(таблица «запрос / правильные статьи / что нашлось» — есть в финальном ноутбуке):

| Тип ошибки (пример) | Мера | Эффект |
|---|---|---|
| Семантический разрыв: «почему дорогая доставка» ↔ статья «Кто оплачивает доставку и сколько она стоит» | bi-encoder каналы C, E | 0.26 → 0.37 → 0.45 |
| Опечатки и разговорные формы: «оюувь», «балы», «Москлвской» | канал char n-грамм | вклад в 0.45 |
| Длинные статьи «размывают» релевантный абзац; статьи с коротким текстом невидимы | чанкование + title в каждом чанке | вклад в 0.37–0.45 |
| Систематические конвенции разметки: десятки формулировок «жалобы на цену доставки» всегда ведут к одной статье, лексически/семантически непохожей на запрос | kNN-канал по выданной разметке | 0.45–0.48 → 0.63 (главный прирост) |
| Тематически близкие статьи «перебивают» правильную | cross-encoder, мягкий бленд рангов | +0.017 |
| Остаток (AP=0 упал 38% → 3.6%) — преимущественно шум разметки: запрос про конкретную книгу → GT «Ограничены звонки и чаты с покупателями» | принят как потолок датасета | — |

## Использованные open-source модели

- [intfloat/multilingual-e5-large](https://huggingface.co/intfloat/multilingual-e5-large) — bi-encoder, ~560M параметров
- [BAAI/bge-m3](https://huggingface.co/BAAI/bge-m3) — bi-encoder, ~568M параметров
- [BAAI/bge-reranker-v2-m3](https://huggingface.co/BAAI/bge-reranker-v2-m3) — cross-encoder, ~568M параметров

Каждая модель < 1B параметров, разворачивается локально; тексты запросов ни в какие внешние сервисы не отправляются.

## Структура репозитория

```
├── README.md
├── pyproject.toml, uv.lock     # окружение (uv)

├── answer.csv                  # финальные предсказания
└── data/                       # articles.f, calibration.f, test.f
    └── cache/                  # эмбеддинги и скоры реранкера — для точного и быстрого воспроизведения
```

## Как запустить

Требуется [uv](https://docs.astral.sh/uv/).

```bash
uv sync                # окружение точно по uv.lock
uv run jupyter lab     # открыть notebooks/final_submission.ipynb, Run All
```

`answer.csv` создаётся в корне; финальная ячейка валидирует формат по всем
требованиям условия. Ноутбук сдан выполненным — метрики и анализ ошибок видны
без запуска.

**Воспроизводимость.** В репозитории лежит `data/cache/` (эмбеддинги статей и
запросов, скоры реранкера): с ним повторный запуск детерминированно выдаёт тот же `answer.csv`, что отправлен на платформу, за несколько минут
на CPU. Полный пересчёт с нуля тоже поддерживается — удалите `data/cache/`
(~30 мин на GPU T4; при первом запуске с HuggingFace скачиваются
веса моделей, ~4.5 ГБ); из-за float-недетерминизма на другом железе единичные
почти-равные кандидаты теоретически могут поменяться местами. Единственный
источник случайности — разбиение фолдов (seed=21).
